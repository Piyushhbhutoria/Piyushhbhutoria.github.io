<p>When I took over the Integrations department, I inherited a dumpster fire. Zero active integrations. A few churned customers. And a codebase so messy that understanding it felt like archaeology.</p><figure><img alt="" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/6edb4ee29db683.png" /></figure><p>The module had no testing framework. It used custom parsing logic that grabbed values, calculated everything in one massive function, and then stuffed them into database-stored templates. Over time, as clients demanded custom logic, everything got crammed into the same file, no structure, no separation of concerns, no variable nomenclature, just an ever-growing ball of complexity.</p><h3>The Hacky Beginning</h3><figure><img alt="Diagram showing client demands and custom logic flowing into a legacy database, creating complexity" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/11bb08aa4b2518.png" /></figure><p>My first task? Build a new integration for a completely different feature in record time. Like any overwhelmed dev, I did what seemed easiest:</p><ol><li>Duplicate everything</li><li>Make minor modifications</li><li>Add a router to pick the right path</li></ol><p>Two critical flaws emerged immediately:</p><ol><li>The original code was already complicated</li><li>The routing logic was based on client business logic, not product features</li></ol><h3>The Problem Statement</h3><p>It took me a full week just to understand how the system worked. The actual integration took several more days. But the real bottleneck? Communication. Explaining to clients how the integration worked, what data we needed, and how our systems would talk to each other.</p><p>Our clients weren’t tech companies. They were manufacturing giants, pharmaceutical firms, paint manufacturers, clothing businesses. Legacy industries with a single tech lead managing all vendor relationships.</p><h3>The Rebuild</h3><p>Once I identified these problems, I knew we needed a complete architectural overhaul. Coming from a developer background, I obsessed over Developer Experience (DX).</p><h3>Phase 1: Kill the complexity</h3><blockquote>Do not reinvent the wheel.</blockquote><p>I ripped out the custom template parsing and replaced it with ERB (Ruby’s native templating engine). This alone made the codebase dramatically easier to work with.</p><h3>Phase 2: Documentation as Code</h3><blockquote>Communication is the key</blockquote><p>I introduced Swagger documentation for all available endpoints. This cut down my back-and-forth with clients significantly and gave them an always-up-to-date reference. No more outdated PDFs floating around emails.</p><p>This was critical because my clients weren’t typical SaaS companies. They were giant legacy businesses manufacturing, pharmaceuticals, paints, clothing. These tech leads understood how to get things done but often worked with implementation partners(we were also one of them) who preferred out-of-the-box solutions and expected us to adapt.</p><p>Every integration became a negotiation to find middle ground.</p><h3>Phase 3: Standardise the Process</h3><blockquote>Structure is essential in building anything that thrives.</blockquote><p>I documented everything:</p><ul><li>Integration requirements and processes</li><li>Data specifications</li><li>Sequential implementation steps</li><li>Typical project timelines</li></ul><p>A standard project flow now looked like this:</p><figure><img alt="" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/32d36d92df29be.png" /></figure><ol><li>Discovery call — understand the client’s business requirements</li><li>Platform mapping — map their needs to our platform’s capabilities</li><li>Solution design — figure out how their ERP works and design the right data flow between systems</li></ol><p>Once solutioning was done, customer support joined for user training while I enabled ERP implementation partners to integrate with our platform.</p><h3>Phase 4: Code Architecture Overhaul</h3><blockquote>If you are afraid to change something, it is clearly poorly designed.</blockquote><p>I restructured the entire codebase using the Factory design pattern:</p><figure><img alt="" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/021df235185371.png" /></figure><ul><li>A base class collated all data points into structured format</li><li>Client-specific logic extended the base class</li><li>Clear separation of concerns between clients</li></ul><p>The trade-off? Templates became code. Every change required a developer who understood ERB. But we solved that later.</p><h3>Phase 5: Testing Framework</h3><blockquote>Quality means doing it right when no one is watching</blockquote><p>I implemented comprehensive tests to ensure internal product changes didn’t break existing integrations. This would also help us in safely migrating our existing clients on to the new architecture.</p><p>In integrations, data integrity is everything. A small mistake can cause operational havoc:</p><ul><li>Lost communication breaks operations</li><li>Incorrect data creates inventory and/or audit issues</li></ul><h3>Phase 6: Migration with Client Collaboration</h3><blockquote>Coming together is a beginning; keeping together is progress; working together is success.</blockquote><p>Started new integrations on the new architecture and slowly developed it to support all previous integrations. Migrating existing integrations to the new architecture was only possible through close client partnerships. There was no way to distinguish between client-specific logic and general logic without asking.</p><p>Client A was instrumental here. We worked together to identify what was truly their requirement versus general integration logic.</p><p>After months of iteration, we reached a milepoint where:</p><ul><li>New developers understood the codebase in one day</li><li>Given clear requirements, a developer could implement a new integration with tests in 2–4 hours</li></ul><p>Tech was no longer the bottleneck. Operations were.</p><h3>Observability: The Missing Piece</h3><p>A critical improvement was enhancing our logging infrastructure.</p><p>When debugging, we constantly needed to check what was sent and received. But clients rarely had logs of incoming requests. We needed logs for every outgoing request.</p><p>I noticed some logs were missing from our logging service. To make it robust, I introduced middleware that logged all inbound and outbound integration traffic.</p><p>Every request to any third-party system was now logged with:</p><ul><li>What we sent</li><li>What we received</li><li>Exact timestamps</li></ul><p>This became an audit trail. We could map requests to entities, understand which services were used by which clients, and run usage queries. It was a game-changer for debugging and analytics.</p><h3>The Self-Service Evolution</h3><p>Our clients didn’t want to do work on their end. They constantly pushed for cosmetic changes on our side that made no sense for general integration patterns.</p><p>Every company implemented their ERP differently. Some used SFTP/FTP to save costs. Others preferred HTTP despite the price.</p><p>The next evolution was building a no-code layer so the Customer Success/implementation team could handle integrations without developers.</p><figure><img alt="" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/49cabe45ffe881.png" /></figure><p>I built a no-code templating engine using the Florric gem. It parsed data inputs and placed values from mappings into custom API payloads. I added a UI showing all available data points for a client. Now the implementation team could build custom API mappings without touching code.</p><p>This also made us safer from cosmetic change requests as every cosmetic request would push integration timeline and/or incur extra cost to clients</p><h3>Future Expansion — Pay-Per-Use Pricing</h3><p>Armed with usage data, we were on track to introduce Pay-Per-Use pricing.</p><p>Previously, integrations were free, one-time setup fee or fixed-cost. Now we could charge based on usage, similar to SAP’s ERP API pricing model. Clients were already used to this from their existing systems.</p><p>The logging infrastructure gave us usage metrics to set fair pricing. We understood usage patterns, behaviors, and which integrations drove the most value.</p><h3>The Outcome</h3><p>Over nine months, the department evolved from dormant with zero clients to:</p><ul><li>9+ active integrations</li><li>Implementation time reduced from days to hours</li><li>Self-serving integration builder for non-technical teams</li><li>Complete audit logging and usage analytics</li><li>Pay-per-use pricing model</li></ul><p>This transformed integrations from a cost center into a scalable revenue stream.</p><figure><img alt="Minimalist chart showing the dramatic reduction in integration implementation time and the growth of active integrations over nine months" src="/blog-media/medium/from-zero-to-hero-building-integration-in-saas-969b38e57240/4067e905ae35e5.png" /></figure><p>Leading integrations taught me that it’s more than just code. It’s cross-functional coordination, customer communication, and relentless iteration. But at the core, it’s about building systems that scale, not just technically, but also operationally.</p><p>If you have made it to the end consider following for more such content.</p>
